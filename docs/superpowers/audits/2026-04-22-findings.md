## Findings

### 1. CRASH — Clicking a name in the Vault throws when the student isn't in the current roster (P0)

**File:** `src/components/modals/StudentProfileModal.jsx:92` (pre-fix line number)

**Repro:**
1. Have any log in the Data Vault.
2. Re-import a roster that doesn't include the student that log references (or have any stale log from a prior session).
3. Click the student's name in the Vault's log row.

**Root cause:** `StudentProfileModal` receives `studentData={allStudents[profileStu]}` from `App.jsx`. When `profileStu` is a studentId not in `allStudents`, `studentData` is `undefined`. Line 92 then runs `migrateIdentity(s).identity` where `s` is undefined — `migrateIdentity(undefined)` returns `undefined`, then `.identity` on `undefined` throws `TypeError: Cannot read properties of undefined (reading 'identity')`.

**Fix:** Top-level export is now a guard wrapper that checks `studentData` before mounting the heavy inner modal. When it's missing, renders a purpose-built `OrphanStudentModal` that explains the situation ("Student not in roster, this log was written for ID X but that student isn't here anymore"). The inner component `StudentProfileModalInner` does all the hooks work. React hooks rules satisfied (no conditional hook calls).

**Prevention:** Added an Escape-key listener via a new `useEscape` hook so orphan modals can always be dismissed. Added a close (×) button in the corner too.

### 2. BLOCKER — Orphan rows in the Vault had no visual differentiation (P1)

**File:** `src/App.jsx` Vault render

**Repro:** same as #1; orphans rendered with their cryptic studentId as the "pseudonym" (e.g. "stu_mr_gone"), indistinguishable from valid students.

**Fix:** The Vault log row now detects orphans (`!allStudents[l.studentId]`), displays them with:
- Leading `↯` indicator
- Muted grey color (not the student's palette color, which doesn't exist)
- Italic
- Tooltip: "Student not in current roster — click for details"
- Truncation for long studentIds

Clicking still opens the orphan modal from fix #1, so no crash — just informative.

### 3. UX HAZARD — No way to reset local data without clearing site storage manually (P2)

**Repro:** Demo data loaded at first visit sticks around. User can't easily "start fresh" to demo the app to another person.

**Fix:** New sidebar action "🧹 Reset Local Data" (red-tinted, below "❓ How it works"). Double-confirms via `window.confirm` with a listed breakdown of what it clears:
- All logs
- All imported students / rosters
- Case memory (incidents / interventions / outcomes)
- Knowledge base documents
- Identity overrides

Explicitly does NOT touch:
- The real-name vault (has its own purge control with separate consent)
- Cloud team data
- Google sign-in session

Implemented by exposing `resetImports()` from `useStudents` and calling it plus `logsBag.setLogs([])`, `caseMemory.clearCaseMemory()`, `kb.setKnowledgeBase([])` from the reset handler.

---

### 4. UX HAZARD — Modals trap users with no keyboard dismissal (P2)

**Repro:** Open the onboarding modal on first visit. Press Escape. Nothing. Click the backdrop — it does close, but Escape is the more expected interaction.

**Fix:** New `src/hooks/useEscape.js` hook. Added to:
- `OrphanStudentModal`
- `OnboardingModal`
- `TeamOnboardingModal` (respects `mustChoose` — can't escape until user picks a team)
- `RealNamesControls` persistence confirmation modal

Surfaced as an audit blocker — Playwright couldn't proceed past a stuck orphan modal because Escape was ignored and the backdrop click couldn't find the close control.

---

### 5. NOISE — Supabase env-not-configured warning on every load (P3)

**Repro:** Load app in dev without `.env.local`. Console prints `[supabase] REACT_APP_SUPABASE_URL / _ANON_KEY not set. Cloud features disabled.` twice (once on initial load, once on reload during audit).

**Deferred:** this is the intended behavior — it's an informational cue for the developer. No user-facing impact; would only be worth muting if we added a "don't warn in prod" flag, which is overkill.
